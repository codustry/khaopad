# Internationalization (i18n)

> **Status: draft** — how Khao Pad handles UI translations vs content translations, and why they live in different layers.

## The key idea: two layers, don't mix them

Khao Pad has **two independent i18n systems** that serve different purposes:

| Layer                    | Tool                      | What it translates                                  | When it compiles     |
| ------------------------ | ------------------------- | --------------------------------------------------- | -------------------- |
| **UI shell**             | Paraglide JS (inlang)     | Buttons, labels, nav, error messages                | Compile time         |
| **Content**              | `article_localizations` etc. in D1 | Article bodies, titles, excerpts — CMS data | Runtime (DB read)    |

They don't overlap. If you find yourself wondering "should this string go in `messages/en.json` or in the article body?" ask: **is this string part of the app, or part of the content an editor wrote?** If the answer is "the app" it's Paraglide. If it's "the author" it's content.

## Layer 1: Paraglide JS (UI shell)

### What it is

Paraglide compiles per-locale JSON into a tree-shakable TypeScript module. Each message becomes a function, so `m.cms_sign_in()` is a regular import with full type safety and dead-code elimination.

```json
// messages/en.json
{ "cms_sign_in": "Sign in", "cms_email": "Email" }

// messages/th.json
{ "cms_sign_in": "เข้าสู่ระบบ", "cms_email": "อีเมล" }
```

```svelte
<script>
  import * as m from '$lib/paraglide/messages';
</script>
<button>{m.cms_sign_in()}</button>
```

Paraglide's Vite plugin (`paraglideVitePlugin` in `vite.config.ts`) regenerates `$lib/paraglide/` on every config change. The output is gitignored — it's a build artifact.

### How it picks the locale at runtime

In a normal SPA Paraglide reads a cookie. In SSR on Workers that's wrong — the same module is shared by all in-flight requests, so the last-cookie-read would leak across requests.

We fix this with `AsyncLocalStorage` in `paraglideLocaleHook`:

```ts
// src/hooks.server.ts
paraglideRuntime.overwriteServerAsyncLocalStorage({ ... })

const paraglideLocaleHook: Handle = async ({ event, resolve }) => {
  const locale = event.locals.locale; // already resolved by bindingsHook
  return paraglideRuntime.serverAsyncLocalStorage.run(
    { locale, origin: event.url.origin, messageCalls: new Set() },
    () => resolve(event),
  );
};
```

Every message call inside `resolve(event)` reads the locale from the AsyncLocalStorage frame instead of the shared runtime. This is the fix for the "switched to Thai but sidebar is still English" bug — without it, the first request's locale stuck.

### Locale resolution priority

`bindingsHook` sets `event.locals.locale` once per request using `localeFromPathname`:

1. First path segment (`/en/...` or `/th/...`) if supported
2. Otherwise `DEFAULT_LOCALE` from `wrangler.toml [vars]` (= `en`)

No cookie, no Accept-Language on the server — URL is the single source of truth. Cookies are for *preference* on the client; the server renders what the URL says.

## Layer 2: Content localizations (CMS data)

### What it is

Per-locale markdown stored in D1. One article row plus one row per locale in `article_localizations`:

```
articles: { id, slug="hello-world", status, ... }
article_localizations:
  { article_id, locale="en", title="Hello world", body="# Hello\n..." }
  { article_id, locale="th", title="สวัสดีชาวโลก", body="# สวัสดี\n..." }
```

The D1 provider hydrates both into `article.localizations = { en, th }` on read.

### How pages pick a localization

```ts
// src/routes/(www)/[locale]/blog/[slug]/+page.server.ts
const localization = article.localizations[locale] ?? article.localizations.en;
```

EN is the canonical fallback (see [CONTENT-MODEL.md](./CONTENT-MODEL.md)). The Paraglide UI stays in the URL's locale — only the article body falls back.

### Why not one huge JSON file?

Two reasons:

1. **Content is frequently edited, separately per locale.** A JSON file means the editor has to edit every locale for every change, or we split by locale — which is where we already are.
2. **Content is queryable.** We filter published articles, search titles, paginate. SQL does this; a JSON blob doesn't.

Paraglide is the opposite — UI strings change rarely, atomically (a new button ships with all its translations), and are enumerated. JSON is correct there.

## Adding a new language

Suppose we want Vietnamese (`vi`). Four places to update:

1. `src/lib/server/content/types.ts` — extend `Locale = 'en' | 'th' | 'vi'`.
2. `project.inlang/settings.json` — add `"vi"` to `locales`.
3. `messages/vi.json` — translate every key in `en.json`.
4. `wrangler.toml [vars]` — `SUPPORTED_LOCALES = "en,th,vi"`.

That's it. Existing articles get no Vietnamese localization; they'll fall back to EN. New articles can optionally add one via the editor (same UX as TH today).

The schema doesn't need a migration — `article_localizations.locale` is already `text`.

## What's `locale` doing in three different places?

It's easy to confuse. Each layer cares for a different reason:

| Where                            | Means                                                |
| -------------------------------- | ---------------------------------------------------- |
| `event.locals.locale`            | Current request's UI locale (for Paraglide)          |
| `params.locale` in route URLs    | Segment of the path the user typed                   |
| `article.localizations[locale]`  | Which body of content to render                      |

Usually they're all equal. They can disagree if you fall back (request is `/th`, but `th` localization is missing, so you render the EN body under a Thai chrome).

## Things we're deliberately **not** doing

- **No locale-in-domain** (`th.khaopad.com`). Subdomains are for role (`www` vs `cms`), not language. Path-based locales keep one deploy, one DNS setup, one cookie scope.
- **No auto-detect via Accept-Language on landing.** The root (`/`) redirects to `/en` (the default). Browsers with Thai preferences will land on `/en` first; we can add a client-side redirect to `/th` later if data shows users want it, but server-side sniffing bakes in guesswork we don't need.
- **No machine translation.** Authors translate. If/when we add AI assists it'll be an editor affordance ("suggest a TH translation"), not automatic content mirroring.
